# 待改善功能清單

## 大量寄信（萬封電子報）架構分析

### 現況問題

目前架構是前端逐筆透過 HTTP 請求呼叫後端寄信，問題如下：
- 寄到一半若瀏覽器關閉或網路中斷，整批任務終止
- 沒有斷點續寄機制，失敗只能重頭來
- 單一 SMTP 連線處理所有信件，無法並發或限速

---

## 待改善項目

### 1. 失敗重寄機制
- 目前已有 `send_logs` 記錄每封信的成功/失敗狀態
- 待實作：介面上新增「重寄失敗信件」按鈕，自動篩選 `status=error` 的列再寄一次
- 難度：低，資料已有，只需前端過濾 + 重送

### 2. 依網域排序寄送
- 大量寄信時，對同一網域（如 `@gmail.com`）短時間送太多封，容易被標記為垃圾郵件或進黑名單
- 待實作：寄信前將收件人清單依 `@` 後的網域分組，輪流寄（每個網域寄一封，再回頭寄下一輪）
- 效果：分散對各 mail server 的請求，降低被封鎖風險
- 難度：中

### 3. 寄發間隔延遲
- 建議：同網域每封之間間隔 1–3 秒，全體每封間隔 0.5–1 秒
- 現況：已有前端逐筆送的迴圈，加上 `await sleep(ms)` 即可
- 分析：若寄 1 萬封、每封延遲 1 秒 = 約 2.8 小時，需要後台任務支撐（見下方）
- 難度：低（小量）/ 高（大量，需後台任務）

### 4. 訊息佇列（Message Queue）
- **是否需要？** → 若需求為「一次寄萬封、可中途離開、斷點續寄」，則必要
- 方案選項：
  - **Redis + Celery**（Python 生態，整合 FastAPI 較成熟）
  - **RQ（Redis Queue）**（更輕量）
  - **簡易 SQLite 任務表**（不引入新服務，適合中小量，如 5000 封以下）
- 目前規模建議：先用「SQLite 任務表 + 後台 worker」方案，夠用且不增加部署複雜度
- 難度：高（架構改動大）

### 5. 後台任務 + 任務狀態追蹤
- 搭配佇列，讓寄信任務在後端背景執行
- 前端定時 polling 任務進度（或 WebSocket 推送）
- 支援暫停 / 繼續 / 取消
- 難度：高

### 6. 大量寄信前的預檢
- 寄送前檢查所有收件人格式、附件是否存在，產生預檢報告再確認
- 現況已有逐列驗證，擴充成批次報告即可
- 難度：低

---

## 優先順序建議

| 優先 | 項目 | 說明 |
|------|------|------|
| ⭐⭐⭐ | 失敗重寄機制 | 資料已有，改動小，效益高 |
| ⭐⭐⭐ | 寄發間隔延遲 | 防封鎖基本措施，小量時簡單實作 |
| ⭐⭐ | 依網域排序寄送 | 中等改動，對大量寄信有效 |
| ⭐ | 訊息佇列 + 後台任務 | 架構改動大，等有萬封需求再評估 |

---

## 結論

目前架構適合 **數百封以內** 的批次寄信。若需求升到萬封等級，需引入後台任務機制，建議屆時再整體評估是否升級為 Celery + Redis 架構。
